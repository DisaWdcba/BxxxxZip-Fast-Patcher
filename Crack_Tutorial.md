## 1. 目标概述

| 项目 | 信息 |
|------|------|
| 目标文件 | `Bandizip.exe` (x64) |
| 版本 | 7.x Standard Edition, Build 71613 |
| 保护机制 | 两级授权验证 (版本类型 + 许可证哈希) |
| 目标效果 | 解除标准版功能限制，显示为企业版已注册 |

Bandizip 的授权模型分为两级：

- **版本类型**（`config.ini` → `config_id`）：控制 `std`（标准版）/ `ent`（企业版）身份
- **许可证哈希**（硬编码值经位运算）：控制 33 处功能开关

两者相互独立但协同工作。只改版本类型而不改许可证哈希，功能限制依然生效。

## 3. 第一阶段：配置系统分析

### 3.1 从字符串入手

IDA 加载 `Bandizip.exe` 后，搜索授权相关字符串。由于 Bandizip 是一个国际化产品，大部分字符串经过编码存储在 `.rdata` 段。先搜索明显的入口点：

```
搜索: RTTI 类名 → CRegisterDlg, CLicenseGuideDlg
搜索: 配置关键字 → SETTING, beta_version, config
```

找到 RTTI 类型描述符 `CRegisterDlg`（注册对话框）和 `CLicenseGuideDlg`（许可证指南对话框），确认存在注册/授权模块。

### 3.2 config.ini 读取入口

通过搜索 `config.ini` 的宽字符串引用，定位到 **`sub_1400E8180`**（配置读取函数）。

**关键反编译代码**：

```c
void sub_1400E8180(__int64 a1)
{
    // 构造 config.ini 路径 (与 exe 同目录)
    // 解析 [setting] 段落
    sub_140044890(ini, L"setting", L"config_id", L"--");

    // 逐级判断版本类型
    if (sub_1400D7AD0(a1, "appx"))         // ← 字符串比较
        *(a1 + 8) = 3;    // 微软商店版
    else if (sub_1400D7AD0(a1, "std"))
        *(a1 + 8) = 1;    // 标准版 (免费/有广告)  ← 补丁点
    else if (sub_1400D7AD0(a1, "ent"))
        *(a1 + 8) = 2;    // 企业版 (付费/无广告)  ← 目标
    else if (sub_1400D7AD0(a1, "GlobalPortable"))
        *(a1 + 8) = 4;    // 便携版
    else
        MessageBoxW(L"Error at config_id...");

    // 同时读取其他配置项
    *(a1 + 12) = GetIniInt(L"beta_version");
    *(a1 + 16) = GetIniInt(L"is_portable");
    // ...
}
```

**关键发现**：

- 版本类型存储在结构体偏移 `+0x08`
- 只有 4 种类型，没有独立的 `Professional`——`ent` 就是专业/企业版
- 该结构体由 `sub_1400E8040()` 返回，全局地址 `xmmword_14031D680`

### 3.3 主初始化流程

**`sub_140138B50`** 是程序主初始化入口，调用顺序：

```c
v11 = sub_1400E8040();       // 获取全局配置结构体
sub_1400E8180((__int64)v11); // 读取 config.ini 填充配置
v12 = sub_140131040();        // 获取全局设置结构体
sub_140131830(v12);           // 设置初始化 (← 隐藏陷阱!)
```

## 4. 第二阶段：许可证哈希系统

### 4.1 发现哈希检查

在分析关于对话框（`CAboutDlg`）的初始化函数 **`sub_1400F2C10`** 时，发现了一处奇怪的位运算：

**反编译代码**：

```c
v11 = sub_140131040();           // 获取设置结构体
v12 = (2 * (v11[72] & 0xD5555555)) | ((v11[72] >> 1) & 0x55555555);
```

这实际上是读取偏移 `0x120`（288 字节处，即 `v11[72]`）的一个 DWORD 值，然后进行位操作。

**追踪来源**——`sub_1401312B0` 中的初始化：

```asm
0x140131541:  mov cs:dword_14031D8B0, 98h   ; 写入 152 (0x98)
```

### 4.2 位运算算法分析

位运算公式：

```python
def hash_transform(v):
    return (2 * (v & 0xD5555555)) | ((v >> 1) & 0x55555555)
```

通过暴力搜索发现这是一对一映射：

```python
>>> hash_transform(152)    # 当前值
100                        # ← 标准版哈希
>>> hash_transform(6944)   # 目标值
10000                      # ← 企业版哈希 (0x2710)
>>> hash_transform(100)
152                        # ← 逆向验证
```

**关键发现**：在 `sub_140138B50` 中存在对哈希值 10000 的检查：

```c
if (hash == 0x2710)        // 10000 = 企业版标识
    beta_version = 0;       // 清除测试版标记
```

### 4.3 33 处检查点

通过搜索 `and ... 0x55555555h` 指令模式，发现整个二进制中有 **33 处**代码使用相同的位运算检查许可证哈希。

这些检查分布在：

| 函数 | 作用 |
|------|------|
| `sub_1400F2C10` | 关于页注册状态显示 |
| `sub_1400F90E0` | 功能开关 |
| `sub_140188E10` | 修复存档对话框限制 |
| `sub_1400E9B70` | 许可证指南对话框（6 种限制消息） |
| ... | ... 等共 33 处 |

这意味着只需修改**源头值**（`dword_14031D8B0`），即可同时影响所有检查。

### 4.4 功能限制消息

**`sub_1400E9B70`** 是许可证指南对话框，根据触发原因 `*(a1 + 72)` 显示不同消息：

| case | 语言文件 ID | 内容 |
|------|-----------|------|
| 0 | `TEXT_LIMIT_PASSMAN` | 密码管理器限制 |
| 1 | `TEXT_SHOW_AD` | 广告显示说明 |
| 2 | `TEXT_CANNOT_CHANGE` | 不可更改 |
| 3 | `TEXT_CANNOT_USE` | **功能不可用（标准版限制）** |
| 4 | `$STATIC_PRICE_DESC2` | 付费功能说明 |
| 5 | `TEXT_PORTABLE_NOTICE` | 便携版限制 |

用户在标准版中使用受限功能时看到的就是 `TEXT_CANNOT_USE`。

## 5. 第三阶段：首次补丁与"完全无效"

### 5.1 第一轮补丁设计

基于以上分析，设计了 5 个补丁：

| # | VA | 修改 | 作用 |
|---|-----|------|------|
| 0a | `0x140131541` | `98`→`1B20` | 许可证 init1 |
| 1 | `0x1400F2D30` | `cmp r12d,64h`→`cmp r12d,r12d` | 注册显示 |
| 2 | `0x1400F338E` | `setz r15b`→`mov r15b,1` | 按钮启用 |
| 3 | `0x1400F33B9` | `cmp r12d,64h`→`cmp r12d,r12d` | 按钮可见 |
| 4 | `0x1400E84BF` | `1`→`2` | 版本类型 |

x64dbg 中应用后，用户反馈：**"完 全 无 效"**

### 5.2 根因定位：x64dbg 内存断点

在 x64dbg 中设置内存写入断点：

```
SetMemoryBreakpoint(dword_14031D8B0, Write)
```

运行后发现断点命中 **两次**：

1. 第一次在 `sub_1401312B0 @ 0x140131541`：写入 `1B20`（已补丁） ✅
2. 第二次在 `sub_140131830 @ 0x140131805`：写入 `0098`（覆盖了！） ❌

**`sub_140131830` 中的第二条初始化指令**：

```asm
0x140131805:  mov dword ptr [rsi+0x120], 98h   ; 再次写入 152
```

### 5.3 教训

**单点 patching 不可靠**。同一内存地址被两个不同初始化函数按顺序写入，只 patch 第一个会被第二个覆盖。必须找到所有写入点。

### 6.1 定位第二条初始化

```
Xrefs to dword_14031D8B0:
```

### 6.2 完整调用链

主初始化 `sub_140138B50` 中的完整调用：


## 7. 第五阶段：补丁逻辑反转

### 7.1 问题表现

补丁 0a、0b、1-4 全部到位后，用户反馈：**"显示 Bandizip 尚未注册"**

### 7.2 代码逻辑分析

重新审视 `sub_1400F2C10` 的控制流：

```asm
0x1400F2D30:  cmp  r12d, 64h         ; hash == 100 ?
0x1400F2D34:  jz   loc_1400F3337      ; 等于→跳过注册数据加载
             ; fall through → 加载注册信息
0x1400F2D3A:  call sub_1400CF570      ; 获取注册数据(email/key)
```

**原始逻辑**：

```
hash == 100  (标准版) → jz 跳转 → 跳过注册数据 → 显示 "未注册"
hash != 100  (企业版 10000) → 不跳转 → 加载注册数据 → 显示 "已注册"
```

**我的补丁 1** 改成了 `cmp r12d, r12d`：

```
cmp r12d,r12d  → 永真 → jz 永远跳转 → 永远跳过注册数据 → 永远显示"未注册"!
```

**补丁逻辑与需求完全相反**。

### 7.3 修正

**撤销补丁 1**，恢复原始 `cmp r12d, 64h`。企业版哈希 10000 ≠ 100，自然不跳转，进入注册信息加载路径。

### 7.4 关于页按钮

企业版哈希 10000 ≠ 100，后续代码也会自然隐藏购买/注册按钮：

```asm
0x1400F33B9:  cmp  r12d, 64h
0x1400F33BD:  jz   loc_1400F33F3    ; 等于100 → 跳过隐藏
              ; fall through → 隐藏按钮
```

增补补丁 2：将 `jz` (0x74 0x34) 改为 `nop; nop` (0x90 0x90)，强制始终执行隐藏逻辑。

### 7.5 教训

**不要盲目 patch 条件跳转**。必须先完整理解控制流的真值表，确认补丁后的行为是否符合预期。

## 8. 第六阶段：文件偏移计算

### 8.1 错误

补丁 2 首次使用了错误的文件偏移 `0xF33BD`（VA 地址），而不是文件偏移。

### 8.2 正确计算

```
.text 段: VA 起始 0x140001000, 文件偏移 0x400
file_offset = (VA - 0x140001000) + 0x400

例如: VA 0x1400F33BD
     = (0x0F33BD - 0x001000) + 0x400
     = 0x0F23BD + 0x400
     = 0x0F27BD
```

### 8.3 完整文件偏移表

| 补丁 | VA | 计算过程 | 文件偏移 |
|------|-----|---------|---------|
| 0a | `0x140131547` | `(131547-1000)+400` | `0x130947` |
| 0b | `0x14013180B` | `(13180B-1000)+400` | `0x130C0B` |
| 1 | `0x1400E84C2` | `(0E84C2-1000)+400` | `0x0E78C2` |
| 2 | `0x1400F33BD` | `(0F33BD-1000)+400` | `0x0F27BD` |

验证方法：用备份文件对比，确认只改变了预期位置。

```powershell
# 验证差异
$orig = [IO.File]::ReadAllBytes("Bandizip.exe.bak")
$patched = [IO.File]::ReadAllBytes("Bandizip.exe")
for ($i=0; $i -lt $patched.Length; $i++) {
    if ($orig[$i] -ne $patched[$i]) { "Diff at $i" }
}
# 输出: 仅 7 个位置有差异，全部符合预期
```

## 9. 第七阶段：Win32 GUI 补丁工具开发

### 9.1 需求

- C 语言 + Win32 原生 API
- GUI 对话框界面（三步操作：检查→补丁→验证）
- 支持"浏览"选择目标目录（不限于当前目录）
- UAC 管理员权限（访问 `C:\Program Files` 等受保护目录）
- Per-Monitor DPI 感知（高分屏不模糊）
- Bandizip 原版图标

### 9.2 技术要点

**DPI 感知**——三级回退策略：

```c
static void InitDPI(void) {
    // Win10 1703+: PerMonitorV2
    SetProcessDpiAwarenessContext(DPI_AWARENESS_CONTEXT_PER_MONITOR_AWARE_V2)
    // Win8.1+: PerMonitor
    SetProcessDpiAwareness(PROCESS_PER_MONITOR_DPI_AWARE)
    // Vista+: System DPI
    SetProcessDPIAware()
}
```

**UAC 清单** (`patcher.manifest`)：

```xml
<requestedExecutionLevel level="requireAdministrator" uiAccess="false"/>
```

编译后通过 `mt.exe` 嵌入：

```bat
mt.exe -manifest patcher.manifest -outputresource:Bandizip-Patcher.exe;1
```

**图标提取**——使用 `pefile` 解析 PE 资源树，提取 RT_GROUP_ICON (ID 20) 的 7 种尺寸图标（16~256px），重打包为标准 `.ico` 文件。

**Unicode API**——全部使用 `W` 后缀 API（`MessageBoxW`, `CreateFileW` 等），避免中文乱码。

### 9.3 项目结构

```
Bandizip-Patcher\
├── Bandizip-Patcher.exe   ← 成品 (152KB)
├── patcher.c              ← 主源代码 (~250 行)
```

## 11. 踩坑总结

### 坑 #1：字符串编码

**现象**：搜索 `config.ini`、`TEXT_CANNOT_USE` 等字符串无结果。

**原因**：Bandizip 使用宽字符串（UTF-16 LE）存储，IDA 默认搜索 ASCII。

**解决**：IDA 中搜索 Wide String 或直接搜索十六进制字节序列 `63 00 6F 00 6E 00 66 00...`（"config.ini" 的 UTF-16 LE 编码）。

### 坑 #2：ark.x64.dll 无授权逻辑

**现象**：怀疑授权验证在 DLL 中，花时间搜索 DLL 内部。

**真相**：`ark.x64.dll` 是纯压缩引擎（LGPL 开源库），不含任何 Bandizip 特定代码。产品密钥 `"a67113f1"` 在 DLL 中根本不存在。所有授权逻辑均在 `Bandizip.exe` 中。

### 坑 #3：双初始化覆盖

**现象**：补丁全部应用后，"完 全 无 效"。

**真相**：`sub_1401312B0` 和 `sub_1401316D0` 按顺序执行，后者覆盖前者的写入。只 patch 第一个白费功夫。

**教训**：对全局变量打补丁前，必须用硬件断点追踪所有写入，确认没有二次覆盖。

### 坑 #4：条件跳转逻辑反转

**现象**：补丁后仍显示"未注册"。

**真相**：`cmp r12d,64h; jz` 的语义是"等于 100 则跳过注册信息加载"。我 patch 成 `cmp r12d,r12d`（永真），反而强制跳过了注册数据，导致永远显示"未注册"。

**教训**：修改条件跳转前，必须画出真值表，确认 target 分支的真实语义。

### 坑 #5：VA 与文件偏移混淆

**现象**：补丁 2 报告"字节不匹配"，实际字节为 `48 89 00 00`。

**真相**：使用了 IDA 的 VA 地址 `0xF33BD` 作为文件偏移。正确文件偏移应为 `0xF27BD`（差 `0x1000 - 0x400 = 0xC00`）。

**教训**：`.text` 段的 VA 基址 ≠ 文件偏移基址。必须使用公式 `file_off = (VA - 0x140001000) + 0x400` 转换。

### 坑 #6：x64dbg dump 的 PE 重建问题

**现象**：`Bandizip.patched.exe`（x64dbg dump）运行报"内存位置访问无效"。

**原因**：x64dbg 的 `dump_module` 重建 PE 时不处理 ASLR 重定位，导致独立运行崩溃。

**解决**：直接对原文件进行文件级二进制补丁，而非依赖 dump。

### 坑 #7：图标格式兼容性

**现象**：`rc.exe` 编译报 `RC2176 : old DIB`。

**原因**：ICO 文件中 DIB 头的 `biHeight` 必须为实际高度的两倍（包含 XOR+AND 掩码），PowerShell 提取的图标未处理此字段。

**解决**：使用 Python `pefile` 库直接解析 PE 资源树，提取 RT_GROUP_ICON（ID 20，主图标组），按标准 ICO 格式重建。

### 经验总结

1. **先画架构图**：理解整体授权模型比盲目 patch 更高效
2. **搜索所有写入点**：对关键全局变量，用交叉引用+内存断点双重验证
3. **真值表先行**：修改条件跳转前，穷举所有分支的输入输出
4. **文件偏移三重确认**：公式计算 + 上下文字节验证 + 差异对比
5. **原位 patching 优于 dump**：直接改原文件避免 PE 重建问题
6. **善用 pefile**：PE 资源提取、图标转换比手工十六进制编辑可靠
